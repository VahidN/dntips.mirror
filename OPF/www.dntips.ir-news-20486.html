‫حفاظت از داده‌ها و افزایش استحکام کد با Array.AsReadOnly در #C


نویسنده: وحید نصیری
تاریخ: ۱۴۰۴/۰۳/۱۱ ۰۹:۱۵
آدرس: www.dntips.ir
در توسعه نرم‌افزار، مدیریت صحیح آرایه‌ها و داده‌های مجموعه‌ای از اهمیت بالایی برخوردار است. یکی از چالش‌های رایج، به اشتراک‌گذاری آرایه‌ها بین بخش‌های مختلف برنامه، مانند کلاس‌ها یا لایه‌های گوناگون، بدون به خطر انداختن یکپارچگی داده‌ها است. کار مستقیم با آرایه‌های داخلی می‌تواند منجر به تغییرات ناخواسته در داده‌های داخلی توسط کدهای خارجی شود و اشکالاتی را ایجاد کند که به‌ویژه در پروژه‌های بزرگ، شناسایی و رفع آن‌ها دشوار است. فریم‌ورک .NET راهکاری ساده و در عین حال قدرتمند برای این منظور ارائه می‌دهد: متد Array.AsReadOnly. این مقاله به بررسی عمیق این ابزار، دلایل سودمندی آن، موارد استفاده پیشرفته و مقایسه آن با گزینه‌های دیگر مانند ImmutableArray می‌پردازد.

آشنایی با Array.AsReadOnly: یک سپر دفاعی برای آرایه‌ها
Array.AsReadOnly یک متد استاتیک (static method) است که یک آرایه موجود را در یک ReadOnlyCollection محصور (wrap) می‌کند و دسترسی فقط-خواندنی (read-only) به عناصر آرایه را فراهم می‌آورد. نکته بسیار مهم این است که Array.AsReadOnly یک کپی جدید از آرایه ایجاد نمی‌کند. در عوض، یک پوشش سریع و بدون هزینه اضافی (zero-cost wrapping) دور آرایه اصلی ایجاد می‌کند. این بدان معناست که عملکرد برنامه تحت تأثیر منفی قرار نمی‌گیرد.
با این حال، باید توجه داشت که اگرچه نسخه محصور شده از طریق رابط فقط-خواندنی، جلوی تغییرات را می‌گیرد، اما اگر هنوز یک ارجاع مستقیم به آرایه اصلی وجود داشته باشد، همچنان می‌توان آن آرایه اصلی را تغییر داد. به عنوان مثال:
int[] numbers = { 1, 2, 3 };
var readOnlyNumbers = Array.AsReadOnly(numbers);

readOnlyNumbers[0] = 10; // خطا: مجموعه فقط-خواندنی است 
numbers[0] = 10; // این کار مجاز است و محتوای readOnlyNumbers نیز تغییر می‌کند
در کد بالا، readOnlyNumbers یک نماینده فقط-خواندنی از آرایه numbers است. تلاش برای تغییر یک عنصر از طریق readOnlyNumbers منجر به خطا در زمان کامپایل یا اجرا می‌شود. اما آرایه اصلی numbers همچنان قابل تغییر است و این تغییرات در readOnlyNumbers نیز منعکس خواهد شد، زیرا هر دو به یک مکان حافظه اشاره دارند.

چرا باید از Array.AsReadOnly استفاده کنیم؟ مزایای کلیدی
استفاده از Array.AsReadOnly در سناریوهای خاصی بسیار ارزشمند است. اصلی‌ترین دلیل، محافظت از آرایه‌های داخلی بدون نیاز به شبیه‌سازی (cloning) پرهزینه است. این ویژگی به شما امکان می‌دهد آرایه‌ها را به طور ایمن بین کلاس‌ها یا لایه‌های مختلف به اشتراک بگذارید. همچنین، به ایجاد رابط‌های برنامه‌نویسی کاربردی (API) تمیز و تغییرناپذیر (immutable) بدون از دست دادن عملکرد کمک می‌کند. به طور خلاصه، Array.AsReadOnly اشتراک‌گذاری ایمن با حداقل سربار عملکردی را فراهم می‌کند. این یک دفاع سریع در برابر تغییرات ناخواسته یا مخرب است.

کاربردهای پیشرفته Array.AsReadOnly در دنیای واقعی
Array.AsReadOnly فراتر از یک ابزار ساده، کاربردهای عملی و پیشرفته‌ای در بهبود استحکام و امنیت کد دارد.

حفاظت از وضعیت داخلی در کتابخانه‌ها و APIها
هنگام ارائه آرایه‌های داخلی از طریق کتابخانه‌ها یا APIها، خطر تغییرات تصادفی یک نگرانی جدی است. به جای بازگرداندن مستقیم آرایه، بهتر است نسخه فقط-خواندنی آن را ارائه دهید.

روش نامناسب:
public class Library
{
    private int[] _data = { 1, 2, 3 }; // داده‌های داخلی
    public int[] GetData() => _data; //دسترسی مستقیم به آرایه 
}
در این روش، کدی که متد GetData را فراخوانی می‌کند، می‌تواند محتوای آرایه _data را تغییر دهد.

روش مناسب:
public class Library
{
    private readonly int[] _data = { 1, 2, 3 }; // داده‌های داخلی و فقط-خواندنی
    public ReadOnlyCollection<int> GetData() => Array.AsReadOnly(_data); 
}
در این روش اصلاح‌شده، مصرف‌کنندگان API می‌توانند داده‌ها را بخوانند اما قادر به تغییر آن نیستند، که به حفظ یکپارچگی داخلی کتابخانه کمک می‌کند.

پیاده‌سازی تغییرناپذیری در طراحی دامنه محور (Domain-Driven Design - DDD)
تغییرناپذیری یک اصل اساسی در DDD برای تضمین سازگاری و پایداری وضعیت تجمعات (aggregates) است. Array.AsReadOnly می‌تواند در این زمینه مفید باشد.
public class Invoice
{
    private readonly LineItem[] _lineItems; // آرایه داخلی برای آیتم‌های فاکتور

    public Invoice(LineItem[] items)
    {
        _lineItems = items ?? throw new ArgumentNullException(nameof(items)); 
    }

    public ReadOnlyCollection<LineItem> LineItems => Array.AsReadOnly(_lineItems);
}
در کلاس Invoice، پس از ایجاد نمونه، لیست LineItems از طریق یک مجموعه فقط-خواندنی در دسترس قرار می‌گیرد. این تضمین می‌کند که پس از ایجاد شیء، هیچ تغییر خارجی در لیست آیتم‌های فاکتور امکان‌پذیر نیست و وضعیت داخلی آن پایدار می‌ماند.

محافظت از داده‌های پیکربندی
مقادیر پیکربندی، مانند فرمت‌های پشتیبانی‌شده، ارائه‌دهندگان پایگاه داده و غیره، نباید در طول زمان اجرای برنامه تغییر کنند. Array.AsReadOnly به محافظت از این داده‌ها کمک می‌کند.
public static class Config
{
    private static readonly string[] _supportedFormats = { "json", "xml", "yaml" }; 
    public static ReadOnlyCollection<string> SupportedFormats => Array.AsReadOnly(_supportedFormats); 
}
با این روش، از رفتارهای غیرقابل پیش‌بینی ناشی از تغییرات پیکربندی در زمان اجرا جلوگیری می‌شود.

دسترسی ایمن در برنامه‌های چندنخی (Multithreaded Applications)
در برنامه‌هایی که چندین نخ (thread) به طور همزمان اجرا می‌شوند، دسترسی مستقیم به آرایه‌های قابل تغییر می‌تواند منجر به شرایط رقابتی (race conditions) و خطاهای پیچیده شود. به جای آن، نسخه‌های فقط-خواندنی از داده‌های اشتراکی را به اشتراک بگذارید.
public class SharedResource
{
    private readonly int[] _sharedData = { 1, 2, 3 }; // داده‌های اشتراکی

    // متدی برای دریافت داده‌های اشتراکی به صورت فقط-خواندنی
    public ReadOnlyCollection<int> GetSharedData() => Array.AsReadOnly(_sharedData); 
}
به این ترتیب، چندین نخ می‌توانند داده‌ها را به طور ایمن و بدون نیاز به قفل‌گذاری (locks) بخوانند، که به بهبود عملکرد و کاهش پیچیدگی کمک می‌کند.

اتصال مجموعه‌های فقط-خواندنی در فریم‌ورک‌های رابط کاربری (UI Frameworks)
در فریم‌ورک‌های UI مانند WPF، Blazor، MAUI یا WinForms، اتصال (binding) لیست‌های قابل تغییر اغلب منجر به اشکالاتی می‌شود که ردیابی آن‌ها دشوار است. استفاده از مجموعه‌های فقط-خواندنی برای اتصال داده به عناصر UI می‌تواند به پایداری بیشتر کمک کند.
public class MenuViewModel
{
    private readonly string[] _menuItems = { "Home", "About", "Contact" }; 
    public ReadOnlyCollection<string> MenuItems => Array.AsReadOnly(_menuItems); 
}
این کار تضمین می‌کند که رابط کاربری و داده‌ها همگام و ایمن باقی بمانند.

مقایسهArray.AsReadOnlyوImmutableArray<T>
گرچه Array.AsReadOnly یک لایه حفاظتی ایجاد می‌کند، اما "تغییرناپذیری واقعی" را تضمین نمی‌کند، زیرا آرایه اصلی همچنان می‌تواند تغییر کند (اگر ارجاعی به آن وجود داشته باشد). در مقابل، ImmutableArray مجموعه‌ای واقعاً تغییرناپذیر است. هرگونه "تغییر" در ImmutableArray منجر به ایجاد یک نمونه جدید از مجموعه با تغییرات اعمال شده می‌شود و نمونه اصلی دست‌نخورده باقی می‌ماند.

جدول زیر تفاوت‌های کلیدی این دو را نشان می‌دهد:

ویژگیArray.AsReadOnly<ImmutableArray<T
عملکردسریع (کپی صفر)کمی کندتر (کپی کردن داده‌ها)
حافظهکمبالاتر (به دلیل ایجاد کپی جدید)
تغییرناپذیری واقعیخیربله
ایمنی در برابر تغییرفقط از طریق پوشش (Wrapper)ایمنی کامل
بهترین مورد استفادهاشتراک‌گذاری داده‌های داخلی کنترل‌شدهAPIهای عمومی که نیاز به تضمین تغییرناپذیری دارند
به طور خلاصه:
  • از Array.AsReadOnly زمانی استفاده کنید که عملکرد اهمیت دارد و داده‌ها به صورت داخلی کنترل می‌شوند.
  • از ImmutableArray<T> زمانی استفاده کنید که به تضمین‌های واقعی تغییرناپذیری برای APIهای عمومی نیاز دارید.

بهترین شیوه‌ها هنگام استفاده از Array.AsReadOnly
برای بهره‌برداری حداکثری از Array.AsReadOnly و جلوگیری از مشکلات احتمالی، رعایت نکات زیر توصیه می‌شود:
  • همیشه ترجیح دهید ReadOnlyCollection<T> را به جای آرایه‌های خام در APIهای عمومی خود ارائه دهید.
  • مراقب باشید: اگر ارجاع به آرایه اصلی به بیرون درز کند (leak)، کدهای خارجی همچنان می‌توانند آن را تغییر دهند. برای داده‌های حیاتی، قبل از ایجاد پوشش فقط-خواندنی، شبیه‌سازی آرایه را در نظر بگیرید.
  • در صورت امکان، آن را با رابط IReadOnlyList<T> ترکیب کنید تا حداکثر انعطاف‌پذیری را داشته باشید.


آزمون کوتاه
کدام یک از گزینه‌های زیر به بهترین شکل تفاوت اصلی Array.AsReadOnly و ImmutableArray را از نظر ایجاد کپی از داده‌ها توصیف می‌کند؟
الف) هر دو یک کپی جدید از آرایه اصلی ایجاد می‌کنند.
ب)Array.AsReadOnly یک کپی جدید ایجاد می‌کند، اما ImmutableArray<T> این کار را نمی‌کند.
ج)Array.AsReadOnly یک کپی جدید ایجاد نمی‌کند (wrapper است)، اما ImmutableArray<T> یک کپی جدید از داده‌ها ایجاد می‌کند.
د) هیچکدام کپی جدیدی از آرایه ایجاد نمی‌کنند.


نتیجه‌گیری
متد Array.AsReadOnly، با وجود سادگی ظاهری، ابزاری بسیار کارآمد در جعبه‌ابزار توسعه‌دهندگان .NET است. این متد با فراهم آوردن امکان اشتراک‌گذاری ایمن آرایه‌ها بدون سربار عملکردی قابل توجه، به ساخت برنامه‌هایی امن‌تر، مقیاس‌پذیرتر و مستحکم‌تر کمک شایانی می‌کند. درک صحیح زمان و نحوه استفاده از Array.AsReadOnly می‌تواند از بروز بسیاری از کلاس‌های باگ، از جمله مشکلات چندنخی و سوءاستفاده تصادفی از API، جلوگیری کند. با به‌کارگیری هوشمندانه این ابزار، می‌توان کیفیت و پایداری کد را به میزان قابل توجهی افزایش داد.


پاسخ آزمون
