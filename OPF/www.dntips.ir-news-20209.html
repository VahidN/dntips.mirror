عنوان:

‫اجتناب از ضد الگوی «آلودگی کد» در توسعه‌ی تست‌های واحد


نویسنده: وحید نصیری
تاریخ: ۱۴۰۴/۰۱/۰۹ ۲۱:۵۰
آدرس: www.dntips.ir
مقدمه
در دنیای توسعه نرم‌افزار، تست واحد (Unit Testing) یکی از ستون‌های اصلی تضمین کیفیت کد است. توسعه‌دهندگان NET. اغلب از تست‌های واحد برای اعتبارسنجی رفتار اجزای کوچک برنامه استفاده می‌کنند. با این حال، گاهی اوقات روش‌های نادرست در نوشتن این تست‌ها می‌تواند به مشکلات جدیدی منجر شود. یکی از این مشکلات، الگوی ضد تست واحد به نام "آلودگی کد" (Code Pollution) است. آلودگی کد زمانی رخ می‌دهد که منطق خاصی فقط برای تسهیل تست در کد اصلی (Production Code) اضافه می‌شود. این کار نه‌تنها پیچیدگی غیرضروری ایجاد می‌کند، بلکه وابستگی بین کد اصلی و تست‌ها را افزایش می‌دهد و نگهداری پروژه را دشوارتر می‌کند. در این مقاله، به بررسی مفهوم آلودگی کد، دلایل وقوع آن، و راهکارهایی برای اجتناب از آن در توسعه NET. می‌پردازیم. هدف این است که توسعه‌دهندگان بتوانند کدهایی تمیزتر، مستقل‌تر و قابل نگهداری‌تر بنویسند.

آلودگی کد چیست و چرا مشکل‌ساز است؟
آلودگی کد به وضعیتی اشاره دارد که در آن، کد اصلی به خاطر نیازهای تست تغییر می‌کند یا با منطق اضافی پر می‌شود. این تغییرات معمولاً شامل اضافه کردن سوئیچ‌ها (Switches)، متدها یا شرط‌هایی است که فقط در محیط تست استفاده می‌شوند. این الگو باعث می‌شود کد اصلی از هدف اصلی خود، یعنی اجرای منطق کسب‌وکار، دور شود و به ابزاری برای پشتیبانی از تست‌ها تبدیل شود.
به عنوان مثال، فرض کنید یک کلاس ساده برای لاگ کردن پیام‌ها در NET. داریم:
public class CustomLogger
{
    private readonly bool _isTestingMode;
    public CustomLogger(bool isTestingMode)
    {
        _isTestingMode = isTestingMode;
    }

    public void Log(string message)
    {
        if (_isTestingMode) return;
        // منطق لاگ کردن پیام در اینجا
    }
}

public class ServiceController
{
    public void Execute(CustomLogger logger)
    {
        logger.Log("Execute method invoked");
    }
}
در این کد، متغیر _isTestingMode و شرط مرتبط با آن فقط برای غیرفعال کردن لاگ در تست‌ها اضافه شده است. این یک نمونه کلاسیک از آلودگی کد است. مشکل اینجاست که این منطق هیچ ارزشی به عملکرد اصلی برنامه اضافه نمی‌کند و تنها برای تست‌ها طراحی شده است. نتیجه؟ کدی که پیچیده‌تر از حد لازم است، وابستگی ناخواسته به محیط تست دارد و در صورت تغییر نیازهای تست، باید بارها بازنویسی شود.

ریشه‌های آلودگی کد
آلودگی کد معمولاً از عدم استفاده صحیح از اصول طراحی نرم‌افزار ناشی می‌شود. یکی از شایع‌ترین دلایل، نادیده گرفتن اصل تزریق وابستگی (Dependency Injection) است. وقتی یک کلاس به‌جای وابستگی به یک abstraction (مانند رابط یا Interface)، مستقیماً به یک پیاده‌سازی خاص وابسته باشد، توسعه‌دهندگان ممکن است برای تست آن، کد اصلی را تغییر دهند. دلیل دیگر، عدم تمایل به بازطراحی کد برای جداسازی نگرانی‌ها (Separation of Concerns) است. در نهایت، فشار زمانی یا عدم آگاهی از روش‌های بهتر نیز می‌تواند به این مشکل دامن بزند.

راهکارهایی برای اجتناب از آلودگی کد
برای رفع این مشکل، باید از تکنیک‌هایی استفاده کنیم که کد اصلی را از منطق تست جدا نگه دارند. در ادامه، چند راهکار کلیدی را بررسی می‌کنیم:

1. استفاده از رابط‌ها (Interfaces) و تزریق وابستگی
به جای افزودن سوئیچ‌های تست در کد اصلی، می‌توانیم از رابط‌ها استفاده کنیم تا وابستگی‌ها را انتزاعی کنیم. برای مثال، کلاس CustomLogger را می‌توان به این شکل بازطراحی کرد:
public interface ILogger
{
    void Log(string message);
}

public class RealLogger : ILogger
{
    public void Log(string message)
    {
        // منطق واقعی لاگ کردن
    }
}

public class FakeLogger : ILogger
{
    public void Log(string message)
    {
        // بدون عملیات، برای تست
    }
}

public class ServiceController
{
    private readonly ILogger _logger;
    public ServiceController(ILogger logger)
    {
        _logger = logger;
    }

    public void Execute()
    {
        _logger.Log("Execute method invoked");
    }
}
در این حالت، ServiceController به یک رابط (ILogger) وابسته است، نه یک پیاده‌سازی خاص. در محیط واقعی، RealLogger تزریق می‌شود و در تست‌ها، FakeLogger استفاده می‌شود. این روش نه‌تنها آلودگی کد را حذف می‌کند، بلکه انعطاف‌پذیری و قابلیت تست‌پذیری (Testability) را افزایش می‌دهد.

2. جداسازی منطق تست از کد اصلی
هرگز نباید کد اصلی را برای تست تغییر داد. اگر نیاز به رفتار خاصی در تست دارید، از ابزارهایی مثل mocking frameworks (مانند Moq در NET.) استفاده کنید. این ابزارها به شما اجازه می‌دهند رفتارهای دلخواه را بدون دستکاری کد اصلی شبیه‌سازی کنید.

3. رعایت اصول SOLID
اصل باز/بسته (Open/Closed Principle) از مجموعه اصول SOLID به ما می‌گوید که باید بتوانیم رفتار یک کلاس را بدون تغییر کد آن گسترش دهیم. استفاده از رابط‌ها و تزریق وابستگی این اصل را محقق می‌کند و نیاز به آلودگی کد را از بین می‌برد.

مزایای اجتناب از آلودگی کد
وقتی از آلودگی کد اجتناب می‌کنیم، کد اصلی تمیزتر و مستقل‌تر می‌شود. این کار نگهداری را آسان‌تر می‌کند، زیرا تغییرات در تست‌ها نیازی به بازنویسی کد اصلی ندارند. همچنین، قابلیت تست‌پذیری افزایش می‌یابد و توسعه‌دهندگان می‌توانند با اطمینان بیشتری refactoring انجام دهند. در نهایت، این رویکرد به کاهش باگ‌های ناخواسته ناشی از منطق تست در محیط واقعی کمک می‌کند.

نتیجه‌گیری
آلودگی کد یک الگوی ضد تست واحد است که می‌تواند کیفیت و نگهداری‌پذیری پروژه‌های NET. را به خطر بیندازد. این مشکل زمانی رخ می‌دهد که منطق تست، به کد اصلی نفوذ کند، مانند افزودن سوئیچ‌ها یا شرط‌های غیرضروری. با استفاده از رابط‌ها، تزریق وابستگی، و ابزارهای مناسب، می‌توان از این الگو اجتناب کرد و کدی تمیز و قابل اعتماد تولید نمود. توسعه‌دهندگان NET. باید به این نکته توجه کنند که هدف تست واحد، اعتبارسنجی رفتار کد است، نه تغییر آن. با رعایت این اصول، نه‌تنها کیفیت کد بهبود می‌یابد، بلکه تجربه توسعه نیز حرفه‌ای‌تر و لذت‌بخش‌تر خواهد شد.
