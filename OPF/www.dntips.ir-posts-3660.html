‫مدیریت شروط اختیاری و مقادیر Null در عبارت‌های LINQ و Entity Framework Core


نویسنده: هادی مزارعی
تاریخ: ۱۴۰۴/۱۰/۰۳ ۱۵:۱۵
آدرس: www.dntips.ir
یکی از چالش‌های رایج در توسعه نرم‌افزارهای مبتنی بر داده، اعمال فیلترهای جستجو بر اساس پارامترهای اختیاری (Optional Parameters) است. پیچیدگی زمانی افزایش می‌یابد که این شروط باید بر روی مجموعه‌های فرزند (Nested Collections) و در عمق گراف اشیاء اعمال شوند. این مقاله با بررسی یک سناریوی واقعی در Entity Framework Core، به نقد رویکردهای مختلف در اعمال فیلترهای تاریخ (Date Range) می‌پردازد و راهکارهای بهینه برای جلوگیری از پیچیدگی کد و بهبود کارایی کوئری‌ها را ارائه می‌دهد.

۱. صورت مسئله و مدل داده
فرض کنید در حال توسعه‌ی سیستم مدیریت پروژه‌های صنعتی هستیم. در این سیستم، هر قطعه (Component) می‌تواند چندین عملیات جفت‌کاری (Fitup) داشته باشد و هر جفت‌کاری شامل چندین عملیات جوشکاری (Weld) است. هر جوش دارای یک گزارش بازرسی (Report) است که تاریخ انجام آن ثبت شده است.
هدف، واکشی قطعاتی است که یکی از شرایط زیر را داشته باشند:
  • هنوز هیچ عملیات جفت‌کاری (Fitup) برای آن‌ها ثبت نشده باشد.
  • جفت‌کاری شده‌اند اما هنوز جوشکاری (Weld) ندارند.
  • جوشکاری شده‌اند و تاریخ گزارش جوش آن‌ها در یک بازه زمانی خاص قرار دارد.

چالش اصلی: بازه زمانی شامل «تاریخ اجرا» (ExecutionDate) و «تاریخ اجرای بعدی» (NextExecutionDate) است. ExecutionDate اجباری است، اما NextExecutionDate اختیاری (Nullable) می‌باشد. اگر NextExecutionDate مقدار نداشته باشد، باید تمام رکوردهای بعد از ExecutionDate بدون سقف زمانی واکشی شوند.

۱-۱. مدل‌های پیشنهادی (Domain Models)
برای درک بهتر کدها، ساختار کلاس‌ها و DbContext به صورت زیر طراحی شده است:

۲. رویکرد اول: استفاده از عملگر سه ‌گانه (Ternary Operator)
در تلاش نخست، برای مدیریت نال‌بودن NextExecutionDate، از عملگر شرطی ? : در داخل عبارت LINQ استفاده شده است.
var query = context.Components.Where(x =>
    !x.Fitups.Any() || // شرط ۱: بدون جفت‌کاری
    (x.Fitups.Any() && x.Fitups.Any(fitup => !fitup.Welds.Any())) || // شرط ۲: بدون جوش
    (command.NextExecutionDate.HasValue ? // شرط ۳: بررسی بازه تاریخ
        (x.Fitups.Any() && x.Fitups.Any(fitup =>
            fitup.Welds.Any(weld =>
                weld.Report.Date >= command.ExecutionDate
                && weld.Report.Date < command.NextExecutionDate.Value))) 
        :
        (x.Fitups.Any() && x.Fitups.Any(fitup =>
            fitup.Welds.Any(weld =>
                weld.Report.Date >= command.ExecutionDate)))
    )
);
نقد فنی رویکرد اول
  • خوانایی پایین (Low Readability): تکرار منطق Fitups.Any(...) در دو بخش true و false شرط، نگهداری کد را دشوار می‌کند.
  • پیچیدگی درخت عبارت (Expression Tree Complexity): اگرچه EF Core در نسخه‌های جدید (3.1 به بعد) در ترجمه شرط‌های پیچیده بهبود یافته است، اما استفاده از عملگرهای شرطی C# در سطح کوئری دیتابیس گاهی منجر به تولید SQL ناکارآمد یا خطای ترجمه (Client-side evaluation exception) در برخی Providerها می‌شود.
  • نقض اصل DRY: منطق فیلتر کردن بر اساس ExecutionDate دو بار تکرار شده است.

۳. رویکرد اصلاح شده: ساده‌سازی با جبر بولی
برای حل مشکل، می‌توان از تکنیک «تخت‌سازی شروط» (Flattening Conditions) با استفاده از جبر بولی استفاده کرد. منطق زیر را در نظر بگیرید:
اگر NextExecutionDate نال باشد، شرط دوم (Date < Next) باید نادیده گرفته شود (همیشه True باشد).
کد اصلاح شده:
var query = context.Components.Where(x =>
    !x.Fitups.Any() 
    || x.Fitups.Any(f => !f.Welds.Any())
    || x.Fitups.Any(f =>
          f.Welds.Any(w =>
             w.Report.Date >= command.ExecutionDate // شرط کف بازه (همیشه ثابت)
             && 
             (!command.NextExecutionDate.HasValue || w.Report.Date < command.NextExecutionDate.Value) // شرط سقف بازه (پویا)
          )
       )
);

تحلیل منطقی (Logic Analysis)
در عبارت (!HasValue || Date < Value) دو حالت رخ می‌دهد:
حالت اول:NextExecutionDate مقدار NULL دارد
در این صورت !HasValue برابر true می‌شود. طبق قوانین منطق (OR Logic)، اگر گزاره اول درست باشد، کل عبارت بدون بررسی گزاره دوم درست ارزیابی می‌شود (Short-circuit).
نتیجه منطقی به شکل زیر در می‌آید:
Result=(Date≥Start)∧(true∨...)
Result=(Date≥Start)∧true
Result=Date≥Start
در جبر بولی، مقدار true عنصر خنثی برای عملگر AND است؛ بنابراین شرط محدودکننده سقف عملاً حذف می‌شود.

حالت دوم:NextExecutionDate دارای مقدار است
در این صورت !HasValue برابر false می‌شود. برای اینکه کل عبارتِ داخل پرانتز true شود، حتماً باید بخش دوم (Date < Value) بررسی شود.
نتیجه منطقی:
Result=(Date≥Start)∧(false∨Date<End)
Result=(Date≥Start)∧(Date<End)
در جبر بولی، مقدار false عنصر خنثی برای عملگر OR است؛ بنابراین شرط دوم مستقیماً اعمال می‌شود.

نکات تکمیلی: ترجمه به SQL و عملکرد دیتابیس
یکی از نکات مهم، نحوه‌ی ترجمه این دستورات به زبان SQL است. درک این موضوع برای بهینه‌سازی Performance حیاتی است.
EF Core الگوی (!HasValue || Column < Value) را معمولاً به الگوی استاندارد SQL زیر ترجمه می‌کند:
WHERE (@p_NextDate IS NULL) OR (ReportDate < @p_NextDate)

۴-۱. مشکل Parameter Sniffing
زمانیکه از این الگو استفاده می‌کنید، دیتابیس (مانند SQL Server) ممکن است در زمان ایجاد Plan اجرایی (Execution Plan) دچار چالش شود.
  • اگر در اولین اجرا پارامتر NULL باشد، دیتابیس فرض می‌کند که اسکن کردن کل جدول بهینه است.
  • اگر در اجرای بعدی پارامتر مقدار داشته باشد (بازه زمانی کوچک)، استفاده از Plan قبلی (Full Scan) برای یک بازه کوچک بسیار کُند خواهد بود.
  • راه‌حل: در سناریوهای بسیار پرفشار، گاهی بهتر است کوئری را به دو شاخه if/else در کد C# تقسیم کنیم تا دو کوئری SQL مجزا تولید شود، یا از هینت‌های دیتابیس (OPTION(RECOMPILE)) استفاده شود. اما برای اکثر کاربردهای معمول، کد اصلاح شده در بخش ۳ کاملاً قابل قبول است.

۴-۲. ایندکس‌گذاری (Indexing)
برای اینکه کوئری فوق کارا باشد، وجود ایندکس بر روی ستون‌های کلید خارجی و ستون تاریخ ضروری است. پیشنهاد می‌شود ایندکس زیر در دیتابیس وجود داشته باشد:
CREATE INDEX IX_Report_Date ON Reports (Date);
-- یا اگر Report داخل Weld است
CREATE INDEX IX_Weld_ReportDate ON Welds (Report_Date);

۵. جمع‌بندی
استفاده از مقادیر تهی (Null) در شروط LINQ نباید منجر به شاخه‌گذاری کد (Branching) با if/else یا عملگرهای ?: شود. با استفاده از الگوی (Parameter IS NULL OR Column == Parameter) می‌توانیم کوئری‌های تمیز، خوانا و قابل ترجمه به SQL استاندارد بنویسیم. در کد نهایی ارائه شده، با حذف تکرارها و استفاده از خواص جبر بولی، منطق انتخاب داده‌ها به شکلی منسجم پیاده‌سازی شد که هم حالت بازه باز (Open-ended range) و هم بازه بسته (Closed range) را به درستی پوشش می‌دهد.
