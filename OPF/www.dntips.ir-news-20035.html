‫هشت مورد از بدترین کارهایی که مایکروسافت با SQL Server انجام داد


نویسنده: وحید نصیری
تاریخ: ۱۴۰۳/۱۱/۳۰ ۰۹:۲۱
آدرس: www.dntips.ir
لیست "بدترین‌ها":
  • شماره ۸: SQL Server روی لینوکس: به دلیل مجوز و تمرکز، "خیلی کم، خیلی دیر" در نظر گرفته شد. نرخ پذیرش پایین.
  • شماره ۷: VARDECIMAL: یک نوع داده کوتاه مدت که در SQL Server 2005 SP2 معرفی و به سرعت منسوخ شد؛ نشان دهنده مدیریت ضعیف محصول.
  • شماره ۶: درون‌ریزی تابع اسکالر: یک ویژگی SQL Server 2019 که مملو از اشکال بود و سال‌ها نیاز به رفع اشکال داشت. مایکروسافت به تدریج این ویژگی را حذف کرده‌است.
  • شماره ۵: سرویس‌های اعلان SQL Server (SSNS): مفهوم ناقص مدیریت اعلان‌های پیچیده در لایه پایگاه داده، که برای وظایفی مانند تلاش‌های مجدد و لغو اشتراک مناسب نیست.
  • شماره ۴: SQL Mail و Database Mail: غیرقابل اعتماد و عیب‌یابی آن دشوار است. باید با یک API برای سرورهای ایمیل یا سرویس‌های خارجی جایگزین می‌شد.
  • شماره ۳: فراخوانی جاوا از رویه‌های ذخیره شده: به طور خاص جاوا، در ویژگی گسترده‌تر Language Extensions؛ غیرضروری و ضعیف اجرا شده.
  • شماره ۲: خوشه‌های داده بزرگ (BDC): بیش از حد مهندسی شده و عملکرد ضعیفی داشت، اساساً نسخه مقیاس‌شده سرورهای مرتبط بود. در نسخه بعدی SQL Server متوقف شد.
  • شماره ۱: ابزارهای توسعه ناپایدار: تغییرات مکرر و رها کردن ابزارهای توسعه (Query Analyzer، SSMS، Data Dude، Azure Data Studio، افزونه mssql در VS Code) مانع از پذیرش توسعه‌دهندگان، یکپارچه‌سازی کنترل منبع و ارتباط مطالب آموزشی شده‌است. این بی‌ثباتی بزرگترین مشکل در نظر گرفته می‌شود.

نتیجه‌گیری: نویسنده از تلاش هدر رفته روی این ویژگی‌ها، ابراز تاسف می‌کند، اما نسبت به آینده SQL Server خوش‌بین است و احساس می‌کند که اضافات اخیر، اگرچه خارق‌العاده نیستند، از مشکلات بزرگ گذشته اجتناب می‌کنند.
