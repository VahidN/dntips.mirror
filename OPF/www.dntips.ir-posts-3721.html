‫توسعه‌ی بومی ویژگی‌های زبان: افزودن اعضای استاتیک سفارشی به کلاس‌های موجود در C# 14


نویسنده: وحید نصیری
تاریخ: ۱۴۰۵/۰۴/۲۹ ۰۸:۲۰
آدرس: www.dntips.ir
زبان سی‌شارپ (#C) همواره با ارائه ویژگی‌های جدید در مسیر بهبود خوانایی کد، افزایش ظرافت نحو (Syntactic Sugar) و ارتقای تجربه توسعه‌دهنده (Developer Experience) گام برداشته است. یکی از چالش‌های دیرینه در طراحی APIها، عدم امکان افزودن اعضای استاتیک (از جمله خواص استاتیک یا Static Properties) به کلاس‌های موجود بدون دستکاری سورس‌کد اصلی آن‌ها بود. در این مقاله، با بررسی یک سناریوی واقعی در مقایسه رشته‌ها (Natural Sort)، راهکارهای گذشته را نقد کرده و نحوه استفاده از بلاک‌های توسعه‌ی جدید (Extension Blocks) در C# 14 را برای گسترش طبیعی‌تر انواع داده در دات‌نت شرح می‌دهیم.

۱. مقدمه و طرح مسئله
در بسیاری از پروژه‌های نرم‌افزاری، نیاز به مقایسه سفارشی رشته‌ها احساس می‌شود. به عنوان مثال، در الگوریتم‌های مرتب‌سازی فایل‌ها، رفتار استاندارد مرتب‌سازی لغوی (Lexicographical Sorting) ممکن است نام رشته‌ها را به صورت زیر مرتب کند:
"file1.txt" -> "file10.txt" -> "file2.txt"
این رفتار از نظر کاربر نهایی منطقی نیست. در مقایسه طبیعی یا Natural Sort Order، انتظار داریم عدد درج‌شده در نام فایل‌ها به عنوان یک مقدار عددی تحلیل شود تا ترکیب به صورت هوشمندانه مرتب گردد:
"file1.txt" -> "file2.txt" -> "file10.txt"
کلاس پایه StringComparer در فریم‌ورک دات‌نت اعضای استاتیکی نظیر StringComparer.Ordinal یا StringComparer.OrdinalIgnoreCase را برای سناریوهای رایج ارائه می‌دهد. اگر بخواهیم الگوریتم NaturalSortComparer اختصاصی خود را بسازیم، پیاده‌سازی پایه آن در C# به شکل زیر خواهد بود:
public class NaturalSortComparer : StringComparer
{
    public override int Compare(string? x, string? y)
    {
        // پیاده‌سازی منطق مقایسه طبیعی (Natural Sort)
        return LogicalStringComparer.Compare(x, y); 
    }

    public override bool Equals(string? x, string? y) => string.Equals(x, y);

    public override int GetHashCode(string obj) => obj?.GetHashCode() ?? 0;
}
چالش اصلی زمانی آغاز می‌شود که می‌خواهیم این کلاس را به گونه‌ای یکپارچه کنیم که توسعه‌دهنده احساس کند NaturalSort بخشی از خود خانواده StringComparer است.

۲. بررسی راهکارهای پیش از C# 14 (رویکرد سنتّی)
پیش از معرفی C# 14، مکانیزم توسعه انواع داده محدود به متدهای توسعه (Extension Methods) بود. برای افزودن این قابلیت به دات‌نت، توسعه‌دهندگان معمولاً از دو الگوی پیاده‌سازی استفاده می‌کردند:
۲.۱. استفاده از متدهای توسعه نمونه (Instance Extension Methods)
در این روش، یک متد توسعه بر روی نمونه‌ای از StringComparer تعریف می‌شد:
public static class StringComparerExtensions
{
    public static StringComparer NaturalSort(this StringComparer comparer) 
        => new NaturalSortComparer();
}
نقاط ضعف این روش:
  • عدم هماهنگی نحو (Syntactic Inconsistency): نحوه فراخوانی به صورت StringComparer.Ordinal.NaturalSort() خواهد بود. ساخت نمونه از یک مقایسه‌گر قبلی جهت فراخوانی مقایسه‌گر جدید غیرضروری و عجیب به نظر می‌رسد.
  • تخصیص حافظه اضافه (Allocation Penalty): اجرای متد روی یک نمونه موجود، خوانایی کد را کاهش داده و نمونه‌سازی‌های بی‌پایه‌ودلیل ایجاد می‌کند.

۲.۲. استفاده از کلاس استاتیک مجزا یا الگوی Singleton
راهکار جایگزین، تعریف یک کلاس مجزا و فراخوانی شیء به صورت Singleton بود:
public static class NaturalSort
{
    public static StringComparer Instance { get; } = new NaturalSortComparer();
}

// نحوه استفاده:
var files = Directory.GetFiles(path)
                     .OrderBy(f => f, NaturalSort.Instance);
اگرچه این رویکرد مشکل نحو ناخوشایند متد توسعه را حل می‌کند، اما نقیصه بزرگ دیگری دارد: عدم کشف‌پذیری (Discoverability). توسعه‌دهنده هنگام تایپ .StringComparer با پیشنهادهای هوشمند کد (IntelliSense) برای این الگوریتم مواجه نمی‌شود و باید از وجود کلاس جداگانه NaturalSort آگاه باشد.

۳. تحول در C# 14: معرفی بلاک‌های توسعه و اعضای استاتیک (Extension Members)
در سی‌شارپ ۱۴، مفهوم متدهای توسعه تکامل یافته و به بلاک‌های توسعه (Extension Blocks) تبدیل شده است. این ویژگی به ما اجازه می‌دهد نه تنها متدها، بلکه خواص استاتیک (Static Properties)، متدهای استاتیک و سایر اعضا را نیز به کلاس‌های موجود اضافه کنیم.

۳.۱. پیاده‌سازی با نحو جدید C# 14
با استفاده از نحو جدید، می‌توانیم خاصیت استاتیک NaturalSort را مستقیماً به خانواده StringComparer تزریق کنیم:
public static class StringComparerExtensions
{
    extension(StringComparer)
    {
        public static StringComparer NaturalSort => new NaturalSortComparer();
    }
}

۳.۲. نحوه استفاده در کد اصلی پروژه
اکنون فراخوانی کد کاملاً طبیعی، خوانا و همگام با اعضای بومی دات‌نت صورت می‌گیرد:
// استفاده کاملاً بومی و حرفه‌ای
var files = Directory.GetFiles(path)
                     .OrderBy(f => f, StringComparer.NaturalSort);
نکته تکمیلی و بهینه‌سازی کارایی (Zero-Allocation Pattern):
در کد اولیه، هر بار فراخوانی StringComparer.NaturalSort یک نمونه جدید از NaturalSortComparer می‌سازد. برای جلوگیری از تخصیص حافظه مداوم و فشار بر روی Garbage Collector، بهتر است نمونه استاتیک را یک‌بار ساخته و Re-use کنیم:
public static class StringComparerExtensions
{
    private static readonly StringComparer _cachedNaturalSort = new NaturalSortComparer();

    extension(StringComparer)
    {
        // بازگرداندن نمونه کش‌شده جهت ارتقای نرخ تخصیص حافظه
        public static StringComparer NaturalSort => _cachedNaturalSort;
    }
}

۵. نتیجه‌گیری
معرفی اعضای استاتیک در Extension Blockهای C# 14 گامی بزرگ در راستای یکپارچه‌سازی و یکدست‌سازی APIها در زبان سی‌شارپ است. این ویژگی به توسعه‌دهندگان اجازه می‌دهد کدهای تمیزتر، یکدست‌تر و با قابل کشف بودن (Discoverability) بالا بنویسند بدون آنکه به نحوه طراحی ابتدایی کلاس اصلی محدود باشند. با بکارگیری خواص استاتیک توسعه‌یافته در کنار الگوهای مدیریت حافظه مانند کشینگ، پروژه‌های دات‌نت از نظر قابلیت نگهداری (Maintainability) و خوانایی به سطح جدیدی ارتقا می‌یابند.
