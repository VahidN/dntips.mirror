‫بهبود عملکرد سریال‌سازی XML در مذاکره محتوا در ASP.NET با استفاده از RecyclableMemoryStream


نویسنده: وحید نصیری
تاریخ: ۱۴۰۴/۰۱/۰۲ ۰۸:۱۰
آدرس: www.dntips.ir
مقدمه
در توسعه برنامه‌های کاربردی وب با استفاده از فریم‌ورک ASP.NET، یکی از جنبه‌های کلیدی که توسعه‌دهندگان با آن مواجه می‌شوند، مدیریت مذاکره محتوا (Content Negotiation) است. این فرآیند به سرور اجازه می‌دهد تا بر اساس درخواست کلاینت، پاسخ را در فرمت‌های مختلفی مانند JSON یا XML ارائه دهد. با این حال، هنگامی که صحبت از سریال‌سازی (Serialization) داده‌ها به فرمت XML می‌شود، عملکرد ممکن است به دلیل استفاده ناکارآمد از حافظه تحت تأثیر قرار گیرد. در این مقاله، به بررسی چگونگی بهبود عملکرد سریال‌سازی XML در ASP.NET با استفاده از ابزار قدرتمند RecyclableMemoryStream می‌پردازیم. این ابزار با مدیریت بهینه حافظه، می‌تواند بار پردازشی را کاهش داده و کارایی برنامه را افزایش دهد. هدف این مطلب، ارائه یک راهنمای آموزشی منسجم برای توسعه‌دهندگان .NET است که به دنبال بهینه‌سازی برنامه‌های خود هستند.


مذاکره محتوا در ASP.NET چیست؟
مذاکره محتوا فرآیندی است که در آن سرور و کلاینت بر سر فرمت داده‌های ارسالی توافق می‌کنند. در ASP.NET، این کار به صورت خودکار توسط فریم‌ورک مدیریت می‌شود. به عنوان مثال، اگر کلاینت در هدر Accept درخواست خود، فرمت application/xml را مشخص کند، سرور تلاش می‌کند پاسخ را در قالب XML ارائه دهد. اما این فرآیند، به ویژه هنگام استفاده از سریال‌سازی XML، می‌تواند به دلیل تخصیص‌های مکرر حافظه و جمع‌آوری زباله (Garbage Collection) سنگین شود.
سریال‌سازی XML به طور پیش‌فرض از کلاس‌هایی مانند XmlSerializer استفاده می‌کند که داده‌ها را به یک جریان حافظه (Memory Stream) تبدیل می‌کنند. این جریان‌ها معمولاً با استفاده از MemoryStream پیاده‌سازی می‌شوند که اگرچه کاربردی است، اما در سناریوهای با بار بالا ممکن است به تخصیص‌های غیرضروری در هیپ اشیاء بزرگ (Large Object Heap - LOH) منجر شود. این موضوع باعث افزایش زمان جمع‌آوری زباله و کاهش عملکرد می‌شود.

معرفی RecyclableMemoryStream
RecyclableMemoryStream یک جایگزین کارآمد برای MemoryStream است که توسط مایکروسافت در بسته Microsoft.IO.RecyclableMemoryStream ارائه شده است. این ابزار با استفاده از یک سیستم استخرسازی (Pooling)، حافظه را به صورت قابل استفاده مجدد مدیریت می‌کند و از تخصیص‌های مکرر در LOH جلوگیری می‌کند. برخلاف MemoryStream که هر بار یک بلوک حافظه جدید تخصیص می‌دهد، RecyclableMemoryStream بلوک‌های حافظه را از یک استخر از پیش تخصیص‌شده بازیافت می‌کند. این رویکرد نه تنها مصرف حافظه را کاهش می‌دهد، بلکه زمان پردازش را نیز بهبود می‌بخشد.
برای استفاده از این ابزار، ابتدا باید بسته NuGet مربوطه را نصب کنید:
Install-Package Microsoft.IO.RecyclableMemoryStream

پیاده‌سازی اولیه سریال‌سازی XML در ASP.NET
بیایید با یک مثال ساده شروع کنیم که نشان‌دهنده سریال‌سازی XML با استفاده از MemoryStream است. فرض کنید یک کنترلر در ASP.NET داریم که لیستی از اشیاء را به فرمت XML برمی‌گرداند:
public class ItemController : ControllerBase
{
    [HttpGet]
    public IActionResult GetItems()
    {
        var items = new List<Item>
        {
            new Item { Id = 1, Name = "Item 1" },
            new Item { Id = 2, Name = "Item 2" }
        };

        var serializer = new XmlSerializer(typeof(List<Item>));
        using var memoryStream = new MemoryStream();
        serializer.Serialize(memoryStream, items);
        memoryStream.Position = 0;

        return File(memoryStream, "application/xml");
    }
}

public class Item
{
    public int Id { get; set; }
    public string Name { get; set; }
}
در این کد، MemoryStream برای ذخیره داده‌های سریال‌شده استفاده می‌شود و سپس به کلاینت بازگردانده می‌شود. اما این روش در برنامه‌های با ترافیک بالا می‌تواند به دلیل تخصیص‌های مکرر حافظه، ناکارآمد باشد.

بهینه‌سازی با RecyclableMemoryStream
اکنون بیایید همین مثال را با استفاده از RecyclableMemoryStream بازنویسی کنیم. ابتدا باید یک نمونه از RecyclableMemoryStreamManager را تعریف کنیم که مسئول مدیریت استخر حافظه است:
public class ItemController : ControllerBase
{
    private readonly RecyclableMemoryStreamManager _streamManager;

    public ItemController()
    {
        _streamManager = new RecyclableMemoryStreamManager();
    }

    [HttpGet]
    public IActionResult GetItems()
    {
        var items = new List<Item>
        {
            new Item { Id = 1, Name = "Item 1" },
            new Item { Id = 2, Name = "Item 2" }
        };

        var serializer = new XmlSerializer(typeof(List<Item>));
        using var recyclableStream = _streamManager.GetStream();
        serializer.Serialize(recyclableStream, items);
        recyclableStream.Position = 0;

        return File(recyclableStream, "application/xml");
    }
}
در این نسخه، به جای MemoryStream از RecyclableMemoryStream استفاده شده است. شیء _streamManager بلوک‌های حافظه را از استخر بازیافت می‌کند و پس از اتمام استفاده، آن‌ها را به استخر بازمی‌گرداند. این کار باعث کاهش فشار بر جمع‌آوری زباله و بهبود عملکرد می‌شود.

مزایای استفاده از RecyclableMemoryStream
  1. کاهش تخصیص در LOH: با استفاده از استخرسازی، از تخصیص‌های بزرگ و پراکنده در حافظه جلوگیری می‌شود.
  2. بهبود عملکرد جمع‌آوری زباله: تعداد دفعات اجرای جمع‌آوری زباله کاهش می‌یابد، زیرا حافظه به صورت مداوم بازیافت می‌شود.
  3. افزایش مقیاس‌پذیری: در برنامه‌هایی با بار بالا، این روش به سرور اجازه می‌دهد درخواست‌های بیشتری را با منابع کمتر پردازش کند.

ملاحظات پیاده‌سازی
برای استفاده بهینه از RecyclableMemoryStream، باید چند نکته را در نظر داشت:
  • تنظیمات استخر: می‌توانید اندازه بلوک‌ها و حداکثر ظرفیت استخر را با استفاده از پارامترهای RecyclableMemoryStreamManager تنظیم کنید. به عنوان مثال:
_streamManager = new RecyclableMemoryStreamManager(
    blockSize: 128 * 1024, // 128 KB
    largeBufferMultiple: 1024 * 1024, // 1 MB
    maximumBufferSize: 16 * 1024 * 1024); // 16 MB
  • مدیریت منابع: همیشه از using استفاده کنید تا مطمئن شوید جریان‌ها به درستی آزاد می‌شوند و به استخر بازمی‌گردند.
  • تست عملکرد: قبل از اعمال تغییرات در محیط تولید، عملکرد را با ابزارهایی مانند BenchmarkDotNet آزمایش کنید.

نتیجه‌گیری
سریال‌سازی XML در مذاکره محتوا یکی از وظایف رایج در توسعه برنامه‌های ASP.NET است، اما می‌تواند به دلیل استفاده ناکارآمد از حافظه، چالش‌هایی را به همراه داشته باشد. با استفاده از RecyclableMemoryStream، توسعه‌دهندگان می‌توانند این فرآیند را بهینه کرده و عملکرد برنامه خود را بهبود بخشند. این ابزار با کاهش تخصیص‌های حافظه و مدیریت هوشمند منابع، به ویژه در سناریوهای با بار بالا، ارزش خود را نشان می‌دهد. توصیه می‌شود که توسعه‌دهندگان .NET این رویکرد را در پروژه‌های خود آزمایش کنند و با تنظیمات مختلف، بهترین نتیجه را برای نیازهای خاص خود به دست آورند. این بهینه‌سازی نه تنها کارایی را افزایش می‌دهد، بلکه تجربه بهتری برای کاربران نهایی فراهم می‌کند.


یک نکته‌ی تکمیلی مهم
استفاده از تزریق وابستگی (Dependency Injection - DI) با طول عمر Singleton برای RecyclableMemoryStreamManager در برنامه‌های ASP.NET معمولاً بهترین رویکرد است، اما این تصمیم به معماری برنامه و نیازهای خاص آن بستگی دارد.

چرا Singleton برای RecyclableMemoryStreamManager مناسب است؟
RecyclableMemoryStreamManager مسئول مدیریت یک استخر حافظه (Memory Pool) است که بلوک‌های حافظه را برای استفاده مجدد ذخیره و بازیافت می‌کند. هدف اصلی این ابزار، کاهش تخصیص‌های مکرر حافظه و بهبود عملکرد است. برای دستیابی به این هدف، موارد زیر را باید در نظر گرفت:
  1. اشتراک استخر حافظه:
    • اگر چندین نمونه از RecyclableMemoryStreamManager ایجاد شود (مثلاً با طول عمر Transient یا Scoped)، هر نمونه یک استخر حافظه جداگانه خواهد داشت. این کار باعث تکه‌تکه شدن حافظه (Fragmentation) و افزایش مصرف حافظه می‌شود، زیرا هر استخر به صورت مستقل بلوک‌های حافظه را تخصیص و مدیریت می‌کند.
    • با استفاده از طول عمر Singleton، یک نمونه واحد از RecyclableMemoryStreamManager در کل برنامه به اشتراک گذاشته می‌شود و تمام درخواست‌ها از یک استخر حافظه مشترک استفاده می‌کنند. این رویکرد کارایی بیشتری دارد و از هدررفت منابع جلوگیری می‌کند.
  2. پایداری حالت استخر:
    • استخر حافظه در RecyclableMemoryStreamManager با استفاده مداوم "گرم" می‌شود، به این معنی که بلوک‌های حافظه از قبل تخصیص داده‌شده و آماده استفاده هستند. اگر هر بار یک نمونه جدید ایجاد شود (مثلاً با Transient)، این مزیت از دست می‌رود و هر نمونه باید از ابتدا استخر خود را بسازد.
    • Singleton تضمین می‌کند که استخر حافظه در طول عمر برنامه پایدار باقی بماند و بهینه عمل کند.
  3. مدیریت منابع:
    • RecyclableMemoryStreamManager به گونه‌ای طراحی شده که نیازی به Dispose کردن ندارد، زیرا خود یک شیء مدیریتی است و جریان‌های ایجادشده توسط آن (مانند RecyclableMemoryStream) را باید Dispose کرد. استفاده از Singleton این اطمینان را می‌دهد که یک نمونه مرکزی بدون نیاز به مدیریت دستی طول عمر، در دسترس باقی بماند.

پیاده‌سازی با تزریق وابستگی
در ASP.NET Core، می‌توانید RecyclableMemoryStreamManager را به عنوان یک سرویس Singleton در کانتینر DI ثبت کنید. این کار معمولاً در فایل Program.cs یا Startup.cs (بسته به نسخه فریم‌ورک) انجام می‌شود.
public class Program
{
    public static void Main(string[] args)
    {
        var builder = WebApplication.CreateBuilder(args);

        // ثبت RecyclableMemoryStreamManager به عنوان Singleton
        builder.Services.AddSingleton<RecyclableMemoryStreamManager>();

        builder.Services.AddControllers();
        var app = builder.Build();
        app.UseRouting();
        app.MapControllers();
        app.Run();
    }
}
سپس می‌توانید آن را در کنترلر خود از طریق سازنده تزریق کنید:
public class ItemController : ControllerBase
{
    private readonly RecyclableMemoryStreamManager _streamManager;

    public ItemController(RecyclableMemoryStreamManager streamManager)
    {
        _streamManager = streamManager;
    }

    [HttpGet]
    public IActionResult GetItems()
    {
        var items = new List<Item> { new Item { Id = 1, Name = "Item 1" } };
        var serializer = new XmlSerializer(typeof(List<Item>));
        using var stream = _streamManager.GetStream();
        serializer.Serialize(stream, items);
        stream.Position = 0;
        return File(stream, "application/xml");
    }
}

public class Item
{
    public int Id { get; set; }
    public string Name { get; set; }
}
موارد استثناء و ملاحظات
اگرچه Singleton معمولاً بهترین انتخاب است، در برخی سناریوها ممکن است نیاز به رویکرد متفاوتی داشته باشید:
  1. چندین استخر با تنظیمات مختلف:
    • اگر برنامه شما نیاز به چندین نمونه از RecyclableMemoryStreamManager با تنظیمات مختلف (مثلاً blockSize یا maximumBufferSize متفاوت) دارد، استفاده از Singleton مناسب نیست. در این صورت، می‌توانید از الگوی Factory یا ثبت چندین سرویس با نام (Named Services) استفاده کنید.
    • مثال با Factory:
builder.Services.AddSingleton<Func<RecyclableMemoryStreamManager>>(_ => 
    () => new RecyclableMemoryStreamManager(
        blockSize: 64 * 1024,
        largeBufferMultiple: 512 * 1024,
        maximumBufferSize: 32 * 1024 * 1024));

public class ItemController : ControllerBase
{
    private readonly RecyclableMemoryStreamManager _streamManager;

    public ItemController(Func<RecyclableMemoryStreamManager> streamManagerFactory)
    {
        _streamManager = streamManagerFactory();
    }
}
  1. کنترل دقیق‌تر طول عمر:
    • در موارد نادر که بخواهید استخر حافظه را در محدوده خاصی (مثلاً یک درخواست خاص) مدیریت کنید، می‌توانید از طول عمر Scoped استفاده کنید. اما این معمولاً توصیه نمی‌شود، زیرا مزایای استخرسازی را کاهش می‌دهد.
  2. تست‌پذیری:
    • اگر برنامه شما نیاز به تست واحد (Unit Testing) دارد و می‌خواهید رفتار RecyclableMemoryStreamManager را در هر تست جدا کنید، ممکن است ترجیح دهید آن را به صورت دستی در تست‌ها ایجاد کنید تا از تزریق Singleton استفاده نکنید.

نتیجه‌گیری
به طور کلی باید RecyclableMemoryStreamManager را از طریق تزریق وابستگی با طول عمر Singleton دریافت کنید، مگر اینکه دلایل خاصی برای استفاده از طول عمر دیگر یا چندین نمونه داشته باشید. این روش با هدف طراحی این ابزار (به اشتراک‌گذاری یک استخر حافظه کارآمد در کل برنامه) هم‌راستا است و بهترین عملکرد را در سناریوهای معمول ASP.NET ارائه می‌دهد. برای اطمینان از کارایی، همیشه تنظیمات استخر را بر اساس نیازهای برنامه خود بهینه کنید و در صورت نیاز به چندین تنظیمات، از الگوهای جایگزین مانند Factory استفاده کنید.
