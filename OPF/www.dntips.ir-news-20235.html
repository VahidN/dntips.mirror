عنوان:

‫روش‌های جایگزین کردن MediatR در پروژه‌های NET.


نویسنده: وحید نصیری
تاریخ: ۱۴۰۴/۰۱/۱۵ ۰۸:۰۴
آدرس: www.dntips.ir
مقدمه
در دنیای توسعه نرم‌افزار با استفاده از فریم‌ورک Microsoft .NET، ابزارها و کتابخانه‌هایی مانند MediatR به دلیل سادگی و انعطاف‌پذیری در مدیریت درخواست‌ها (Requests) و پاسخ‌ها (Responses) در الگوی CQRS (Command Query Responsibility Segregation) و معماری مبتنی بر واسطه (Mediator Pattern) بسیار محبوب شده‌اند. با این حال، در برخی پروژه‌ها ممکن است نیاز به جایگزینی MediatR با روش‌ها یا ابزارهای دیگر احساس شود؛ چه به دلیل پیچیدگی‌های غیرضروری، نیاز به عملکرد بهتر، یا تمایل به استفاده از راهکارهای توزیع‌شده (Distributed Systems). در این مقاله، به بررسی گزینه‌های جایگزین برای MediatR در پروژه‌های .NET می‌پردازیم و با تحلیل مزایا و معایب هر روش، راهنمایی جامعی برای توسعه‌دهندگان ارائه می‌دهیم. هدف این است که شما بتوانید با توجه به نیازهای پروژه خود، بهترین تصمیم را بگیرید.

1. تزریق مستقیم وابستگی‌ها (Direct Dependency Injection)
یکی از ساده‌ترین روش‌های جایگزین برای MediatR، استفاده از تزریق مستقیم وابستگی‌ها (Dependency Injection) است که به صورت پیش‌فرض در فریم‌ورک .NET Core ارائه می‌شود. در این روش، به‌جای استفاده از یک واسطه مرکزی مانند MediatR برای ارسال و مدیریت درخواست‌ها، سرویس‌ها و وابستگی‌ها مستقیماً در کلاس‌هایی که به آن‌ها نیاز دارند تزریق می‌شوند.

مزایا:
  • کاهش پیچیدگی: حذف لایه واسطه باعث می‌شود کد خواناتر و قابل‌فهم‌تر شود.
  • عملکرد بهتر: عدم نیاز به پردازش درخواست‌ها از طریق یک واسطه، زمان اجرا را کاهش می‌دهد.
  • کنترل بیشتر: توسعه‌دهندگان می‌توانند به صورت مستقیم با سرویس‌ها تعامل داشته باشند.
معایب:
  • افزایش وابستگی‌ها: در پروژه‌های بزرگ، تعداد وابستگی‌های تزریق‌شده ممکن است مدیریت کد را دشوار کند.
  • عدم جداسازی مسئولیت‌ها: برخلاف MediatR که به طور ذاتی از الگوی CQRS پشتیبانی می‌کند، در این روش باید این جداسازی به صورت دستی پیاده‌سازی شود.

مثال کد:
public class OrderController : ControllerBase
{
    private readonly IOrderService _orderService;

    public OrderController(IOrderService orderService)
    {
        _orderService = orderService;
    }

    [HttpPost]
    public IActionResult CreateOrder(OrderDto order)
    {
        _orderService.CreateOrder(order);
        return Ok();
    }
}
در این مثال، به جای استفاده از MediatR برای ارسال یک دستور (Command)، سرویس موردنظر مستقیماً تزریق شده و فراخوانی می‌شود.

2. MassTransit: چارچوبی برای سیستم‌های توزیع‌شده
MassTransit یک کتابخانه قدرتمند و متن‌باز (Open-Source) است که برای توسعه برنامه‌های توزیع‌شده طراحی شده و قابلیت‌هایی مانند ارسال پیام (Messaging)، مدیریت رویدادها (Event Handling) و پشتیبانی از صف‌های پیام (Message Queues) مانند RabbitMQ و Azure Service Bus را ارائه می‌دهد. این ابزار می‌تواند جایگزین مناسبی برای MediatR در پروژه‌هایی باشد که نیاز به ارتباط بین سرویس‌ها یا معماری میکروسرویس دارند.

مزایا:
  • پشتیبانی از معماری توزیع‌شده: MassTransit برای سناریوهایی که نیاز به ارسال پیام بین سرویس‌ها یا سیستم‌ها دارند، ایده‌آل است.
  • انعطاف‌پذیری: امکان استفاده از پروتکل‌های مختلف و ابزارهای پیام‌رسانی را فراهم می‌کند.
  • قابلیت توسعه: ویژگی‌هایی مانند Saga برای مدیریت تراکنش‌های طولانی‌مدت (Long-Running Transactions) را دارد.
معایب:
  • پیچیدگی بیشتر: نسبت به MediatR، راه‌اندازی و پیکربندی آن زمان‌برتر است.
  • مناسب نبودن برای پروژه‌های کوچک: اگر پروژه شما نیازی به معماری توزیع‌شده ندارد، استفاده از MassTransit ممکن است بیش از حد پیچیده باشد.

مثال کد:
public class OrderConsumer : IConsumer<CreateOrderCommand>
{
    public Task Consume(ConsumeContext<CreateOrderCommand> context)
    {
        // منطق پردازش سفارش
        return Task.CompletedTask;
    }
}

// پیکربندی MassTransit
services.AddMassTransit(x =>
{
    x.AddConsumer<OrderConsumer>();
    x.UsingRabbitMq((context, cfg) =>
    {
        cfg.Host("localhost");
        cfg.ConfigureEndpoints(context);
    });
});
در اینجا، MassTransit به عنوان یک سیستم پیام‌رسان عمل می‌کند و دستورات (Commands) را به مصرف‌کننده‌ها (Consumers) ارسال می‌کند.

3. CAP: راهکاری سبک برای رویدادها
CAP (مخفف C# Advanced Publisher) یک کتابخانه سبک و کارآمد است که برای انتشار و اشتراک رویدادها (Event Publishing and Subscribing) در سیستم‌های توزیع‌شده طراحی شده است. این ابزار از الگوی رویدادمحور (Event-Driven) پشتیبانی می‌کند و با ابزارهایی مانند Kafka و Redis سازگار است.

مزایا:
  • سادگی: نسبت به MassTransit، راه‌اندازی و استفاده از CAP ساده‌تر است.
  • عملکرد بالا: به دلیل سبک بودن، برای پروژه‌هایی با حجم تراکنش بالا مناسب است.
  • قابلیت اطمینان: از مکانیزم‌هایی مانند "حداقل یک‌بار تحویل" (At-Least-Once Delivery) پشتیبانی می‌کند.
معایب:
  • محدودیت در قابلیت‌ها: نسبت به MassTransit، امکانات کمتری برای مدیریت پیچیدگی‌های پیشرفته دارد.
  • تمرکز بر رویدادها: بیشتر برای سناریوهای رویدادمحور مناسب است تا دستورات (Commands).

مثال کد:
public class OrderController : ControllerBase
{
    private readonly ICapPublisher _capPublisher;

    public OrderController(ICapPublisher capPublisher)
    {
        _capPublisher = capPublisher;
    }

    [HttpPost]
    public async Task<IActionResult> CreateOrder(OrderDto order)
    {
        await _capPublisher.PublishAsync("order.created", order);
        return Ok();
    }
}
در این مثال، CAP برای انتشار یک رویداد استفاده شده که می‌تواند توسط سرویس‌های دیگر اشتراک شود.

4. Brighter: پردازشگر دستورات مشابه MediatR
Brighter یک کتابخانه دیگر است که الگوی پردازش دستورات (Command Processor) و ارسال (Dispatching) را پیاده‌سازی می‌کند و شباهت زیادی به MediatR دارد. این ابزار بر سادگی و عملکرد تمرکز دارد و قابلیت‌هایی مانند زنجیره مسئولیت (Chain of Responsibility) را ارائه می‌دهد.

مزایا:
  • شباهت به MediatR: اگر با MediatR آشنا هستید، انتقال به Brighter آسان خواهد بود.
  • قابلیت‌های اضافی: امکاناتی مانند Retry و Circuit Breaker را به صورت داخلی پشتیبانی می‌کند.
  • انعطاف‌پذیری: می‌تواند با سیستم‌های پیام‌رسان مانند RabbitMQ ادغام شود.
معایب:
  • جامعه کاربری کوچکتر: نسبت به MediatR، منابع و پشتیبانی کمتری دارد.
  • پیچیدگی در سناریوهای خاص: ممکن است برای پروژه‌های بسیار بزرگ به اندازه کافی مقیاس‌پذیر نباشد.

مثال کد:
public class CreateOrderCommand : IRequest
{
    public OrderDto Order { get; set; }
}

public class CreateOrderHandler : RequestHandler<CreateOrderCommand>
{
    public override void Handle(CreateOrderCommand command)
    {
        // منطق ایجاد سفارش
    }
}

// پیکربندی Brighter
var dispatcher = new CommandDispatcher();
dispatcher.Dispatch(new CreateOrderCommand { Order = order });
این کد نشان می‌دهد که Brighter چگونه می‌تواند دستورات را مشابه MediatR مدیریت کند.

نتیجه‌گیری
انتخاب جایگزین مناسب برای MediatR به نیازهای خاص پروژه شما بستگی دارد. اگر به دنبال سادگی و عملکرد بالا هستید، تزریق مستقیم وابستگی‌ها گزینه‌ای عالی است. برای پروژه‌های توزیع‌شده و پیچیده‌تر، MassTransit و CAP ابزارهای قدرتمندی هستند که هر کدام نقاط قوت خود را دارند: MassTransit برای سناریوهای پیشرفته‌تر و CAP برای راهکارهای سبک‌تر. در نهایت، Brighter می‌تواند گزینه‌ای میانی باشد که تعادلی بین سادگی MediatR و قابلیت‌های اضافی ارائه می‌دهد.
توصیه می‌شود پیش از انتخاب، نیازمندی‌های پروژه (مانند مقیاس‌پذیری، پیچیدگی و نیاز به معماری توزیع‌شده) را به دقت بررسی کنید و با آزمایش هر ابزار در یک پروژه نمونه، بهترین گزینه را انتخاب کنید. این رویکرد به شما کمک می‌کند تا ضمن حفظ کیفیت کد، بهره‌وری تیم توسعه را نیز افزایش دهید.
